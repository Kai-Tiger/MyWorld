# 立项前需求分析文档

> 项目：浏览器端黑暗奇幻 3D 动作探索游戏原型  
> 日期：2026-06-18  
> 依据：当前代码库、`conversation-history.md`、`docs/requirements/`、资源目录与现有实现状态

## 1. 文档目的

本文从需求工程角度，对当前项目进行立项前分析，目标是回答以下问题：

- 这个项目要解决什么体验目标？
- 当前已经具备哪些能力，距离可立项的垂直切片还缺什么？
- 哪些需求属于核心范围，哪些应该延后？
- 从技术、内容生产、性能和体验角度看，主要风险是什么？
- 如果正式立项，第一阶段应该交付什么，并用什么标准验收？

本文不是代码实现方案，也不是逐项开发任务清单。它用于在正式立项前统一产品目标、范围边界、优先级和验收口径。

## 2. 项目背景与现状

### 2.1 起点

项目最初是一个基于 Three.js + Vite 的浏览器端等轴测 3D Demo，包含：

- 玩家在室外地图移动。
- 简单房屋、NPC、地图和碰撞。
- 进入室内/离开室内的基础场景切换。

### 2.2 当前演进方向

经过多轮迭代，项目已经明显从“等轴测休闲 Demo”演进为“黑暗奇幻动作探索原型”：

- 视觉参考转向黑暗之魂式暗黑、城堡、雾门、峡谷、废墟。
- 玩家从简单移动扩展到第三人称相机、攻击、连击、格挡、跳跃、法术、MP、道具与武器切换。
- 世界从简单村庄扩展为出生点、城堡、峡谷、旧教堂废墟、曲线峭壁、巡逻敌人等可探索区域。
- 资产生产从程序化几何扩展到 Blender 建模、GLB/FBX 导入、碰撞导出、模型优化和图标渲染。

### 2.3 当前项目状态判断

当前项目处于“高频实验后的原型聚合阶段”：

- 已有多个关键玩法系统的雏形。
- 已有大量真实资产和 Blender 导出流程。
- 已暴露出典型 3D 游戏工程风险：碰撞精度、资源性能、镜头遮挡、视觉一致性、动作手感、场景状态管理。
- 当前 README 与实际项目方向存在偏差，需要后续更新。

立项前最重要的工作不是继续扩功能，而是收敛为一个明确的垂直切片。

## 3. 产品愿景

### 3.1 一句话定位

一个运行在浏览器中的黑暗奇幻 3D 动作探索原型，玩家从村庄出发，穿过曲折地形和废墟，进入多层城堡，使用近战、法术和闪避/防御式走位对抗敌人，并通过篝火和探索推进体验。

### 3.2 目标体验

项目希望提供的核心体验是：

- 黑暗奇幻氛围：阴暗但可读的环境、城堡、雾门、废墟、火把和峡谷。
- 空间探索：从出生点到城堡的曲折路线，多层室内结构和可探索遗迹。
- 轻量动作战斗：锁定敌人、近战连击、法术、MP、敌人仇恨和归巢。
- 连贯场景体验：进入城堡、搭乘电梯、死亡淡出、复活和加载预热尽量减少突兀感。

### 3.3 立项价值

该项目适合立项为一个“技术与玩法垂直切片”，原因：

- 浏览器端 3D 动作探索具有展示价值和技术挑战。
- 已经积累了较多可复用资产、系统和经验。
- 当前问题具有明确工程边界，可以通过范围收敛形成可验收 Demo。
- 若继续无边界扩展，会快速陷入资产、性能和系统复杂度失控。

## 4. 用户与使用场景

### 4.1 目标用户

第一阶段目标用户不是大众玩家，而是：

- 项目作者/开发者：验证技术路线和玩法方向。
- 内部试玩者：体验垂直切片，反馈操作、视觉和性能问题。
- 潜在合作者：判断项目是否具备继续开发价值。

### 4.2 核心使用场景

1. 玩家打开浏览器进入游戏。
2. Loading 阶段完成模型预加载和渲染预热。
3. 玩家从出生点附近出发，熟悉移动、相机和基础 UI。
4. 玩家经过营火、NPC 或敌人布置区域。
5. 玩家沿曲折路线或峡谷探索至城堡/废墟。
6. 玩家与敌人战斗，使用近战、连击和火球。
7. 玩家进入城堡，体验雾门、室内、多层结构、电梯和屋顶。
8. 玩家死亡后看到敌人与机关继续运行至黑屏，然后从篝火/检查点复活。

## 5. 范围定义

### 5.1 建议立项范围：垂直切片

建议第一阶段只立项为一个可玩的垂直切片，而不是完整游戏。

垂直切片范围：

- 一个完整室外起点区域。
- 一条通往城堡的可控探索路线。
- 一个旧教堂废墟探索点。
- 一个多层城堡探索点。
- 少量敌人、巡逻和战斗。
- 一个篝火/复活循环。
- 一套基础 UI、装备栏和 MP/HP 显示。
- 基础音效、加载预热和性能可接受的资源管线。

### 5.2 明确不在第一阶段范围

以下内容不建议纳入第一阶段正式范围：

- 完整关卡流程和剧情章节。
- 大量武器、装备、技能树和道具系统。
- 完整 NPC 分支对话和任务系统。
- 存档系统、设置菜单、键位重绑。
- 多平台适配，尤其是移动端完整操作体验。
- 联机、区块链、账号系统。
- 大规模开放世界。
- 商业级美术资产统一重制。

## 7. 功能需求

### 7.1 玩家控制与相机

#### FR-PLAYER-001 移动与基础控制

玩家应能在室外、废墟和城堡场景中使用键盘移动。

验收标准：

- WASD/方向键可控制角色移动。
- 玩家移动不穿过主要墙体、山体、城堡结构和废墟墙体。
- 场景切换或死亡复活后控制状态恢复正常。

#### FR-PLAYER-002 第三人称相机

相机应支持水平和垂直视角调整，并在负 pitch 时动态拉近，避免从角色脚底观看。

验收标准：

- 鼠标拖动可调整视角。
- pitch 被限制在配置范围内。
- 负 pitch 时相机距离缩短，角色不会被异常仰视穿模。
- 相机遮挡检测不造成明显卡顿。

#### FR-PLAYER-003 跳跃与地形适配

玩家应能使用跳跃动画和物理在允许区域内移动，但不能通过跳跃越过关键边界。

验收标准：

- 空格触发 jump 动画。
- 跳跃保留重力和落地逻辑。
- 峭壁、城堡边界、废墟墙体不应被跳跃绕过。

### 7.2 战斗系统

#### FR-COMBAT-001 近战攻击与连击

玩家应能使用近战攻击，并支持两段连击。

验收标准：

- 按 J 触发第一段攻击。
- 合理窗口内再次按 J 触发第二段攻击。
- 第二段攻击拥有独立伤害、范围和前冲手感。
- 攻击命中窗口与动画表现基本一致。

#### FR-COMBAT-002 法术与 MP

玩家应能消耗 MP 释放火球。

验收标准：

- 火球消耗配置化 MP。
- MP 不足时不播放施法动画、不进入冷却、不生成火球。
- 火球命中敌人造成伤害并触发仇恨。
- 篝火休息和死亡复活恢复 MP。

#### FR-COMBAT-003 防御与体力

玩家应具备基础体力和防御资源逻辑。

验收标准：

- 攻击和防御消耗体力。
- 体力不足影响动作或移动。
- 防御命中反馈与音效可读。

### 7.3 敌人 AI

#### FR-AI-001 感知与追击

敌人应能发现玩家并追击。

验收标准：

- 玩家进入触发范围后敌人进入追击。
- 远程命中敌人也会触发追击。
- 敌人攻击有冷却、攻击窗口和转向限制。

#### FR-AI-002 归巢与重新激活

敌人被拉出过远时应返回初始点，但返回途中可以被重新激活。

验收标准：

- 超过归巢半径后敌人返回初始位置。
- 返回使用 walk 动画。
- 返回途中被攻击或看到玩家会重新追击。
- 回到初始点后恢复原始朝向或巡逻状态。

#### FR-AI-003 巡逻敌人

部分敌人应能沿两个或多个点巡逻，并在发现玩家后切换到战斗。

验收标准：

- 巡逻敌人使用 walk 动画移动。
- 发现玩家后进入攻击/追击。
- 脱离仇恨后继续巡逻，而不是永久坐下或停滞。

### 7.4 场景与关卡

#### FR-WORLD-001 室外出生区域

室外出生区域应承担教学和引导作用。

验收标准：

- 玩家出生后能看到基础环境、篝火或引导目标。
- 路线能自然引向城堡或废墟。
- 不出现明显地形穿模、漂浮模型或不可解释阻挡。

#### FR-WORLD-002 城堡

城堡应是第一阶段的主要探索目标。

验收标准：

- 城堡外观、入口和白雾门位置正确。
- 玩家可进入城堡并在多层空间中探索。
- 电梯可在一楼和屋顶之间按规则运行。
- 屋顶、楼层、楼梯和中庭碰撞可靠。
- 城堡资源可通过 Blender 导出流程更新到游戏。

#### FR-WORLD-003 旧教堂废墟

旧教堂废墟应作为可进入探索点，而非纯装饰模型。

验收标准：

- 玩家可以进入废墟内部。
- 碰撞体按模型内部结构生成，不使用整栋大盒子阻挡。
- 优化模型在浏览器中加载和转向时不卡顿到不可接受。

#### FR-WORLD-004 峡谷与峭壁

地图应包含不可攀登的高陡峭壁，形成空间边界和路线塑形。

验收标准：

- 曲线峭壁视觉连续。
- 峭壁能够阻挡玩家攀爬或穿越。
- 峡谷路线保留可通行中轴。

### 7.5 交互与 UI

#### FR-UI-001 基础 HUD

HUD 应显示玩家关键状态。

验收标准：

- 左上角显示 HP。
- HP 下方显示 MP。
- 状态条尺寸稳定，不因数值变化抖动。

#### FR-UI-002 装备栏

左下角装备栏应提供类似 Dark Souls 的四向装备展示。

验收标准：

- 菱形布局展示法术、盾牌、武器、道具。
- 锤子可作为武器切换和显示。
- 图标和模型状态一致。

#### FR-UI-003 场景交互提示

靠近可交互对象时应出现清晰提示。

验收标准：

- 进入城堡、离开城堡、篝火休息、电梯开关等交互有提示。
- 提示位置不遮挡核心视野。
- 不再依赖移动端虚拟方向键。

### 7.6 资产与工具链

#### FR-PIPE-001 Blender 到游戏导出

城堡和复杂场景应通过 Blender 维护，并通过脚本导出到游戏。

验收标准：

- `castle_master.blend` 是城堡主资产来源。
- 导出脚本生成 GLB 分区和 manifest。
- 碰撞、socket、lever 等游戏语义数据可被导出和加载。

#### FR-PIPE-002 模型优化与碰撞导出

大型 GLB 应具备优化和独立碰撞生成流程。

验收标准：

- 旧教堂等大型资源有优化脚本。
- 视觉模型与碰撞数据可分离。
- 碰撞 JSON 能按游戏位置、旋转和缩放正确应用。

#### FR-PIPE-003 模型图标生成

装备模型应能渲染为 2D UI 图标。

验收标准：

- 可从 GLB/FBX 等 3D 模型生成透明 PNG 图标。
- 图标居中、裁切合理，在装备栏中可读。

## 8. 非功能需求

### 8.1 性能

#### NFR-PERF-001 启动性能

首次进入游戏应通过模型预加载和预渲染降低明显卡顿。

目标：

- Loading 阶段预加载核心模型。
- 对大模型所在方向进行隐藏预渲染。
- Loading 结束后玩家首次转向废墟/城堡时不出现明显长卡顿。

#### NFR-PERF-002 运行帧率

第一阶段目标不是商业级高帧率，但应达到可试玩。

建议目标：

- 桌面 Chrome 中普通区域稳定接近 60 FPS。
- 面向城堡/废墟等重区域时不应出现连续卡顿。
- 单次资源编译卡顿应尽量隐藏在 loading 阶段。

### 8.2 可维护性

#### NFR-MAINT-001 配置化

战斗、敌人、世界布局、光照和城堡资源版本应尽量配置化。

目标：

- 不通过硬编码散落常量调整核心玩法。
- 核心数值集中在 `src/config/`。
- 世界布置可在配置中快速调整。

#### NFR-MAINT-002 需求文档化

正式计划应沉淀为独立需求文档。

目标：

- 新计划写入 `docs/requirements/`。
- 聊天历史保留在 `conversation-history.md`。
- 需求文档用于后续实现和验收。

### 8.3 可靠性

#### NFR-REL-001 碰撞可靠性

碰撞是当前项目最大体验风险之一。

目标：

- 玩家不应穿过主要场景结构。
- 玩家不应在电梯、楼层、屋顶、废墟等关键区域掉出地图。
- 隐藏碰撞体不应阻挡合理路线。

#### NFR-REL-002 状态恢复

死亡、复活、场景切换后状态应可恢复。

目标：

- 玩家死亡后能从篝火/检查点复活。
- 敌人和城堡机关按设计重置或保持。
- UI、相机、输入状态不残留异常。

### 8.4 视觉一致性

#### NFR-VIS-001 黑暗但可读

画面应保持黑暗奇幻氛围，但暗处不能不可读。

目标：

- 亮部不过曝。
- 暗部可辨认路线和敌人。
- 室内、夜晚、峡谷和废墟的亮度策略一致。

## 9. MVP 建议

### 9.1 MVP 定义

建议第一阶段 MVP 是一个 10 到 15 分钟可试玩垂直切片。

玩家能完成：

1. 从出生点出现。
2. 查看 HP/MP 和装备栏。
3. 与篝火或 NPC 附近区域接触。
4. 沿曲折路线前往废墟或城堡。
5. 与至少 2 类敌人战斗。
6. 使用近战连击和火球。
7. 进入城堡，通过电梯到屋顶。
8. 死亡后复活，状态恢复正常。

### 9.2 MVP 必须项

- 稳定加载和运行。
- 玩家移动、相机、碰撞可用。
- 近战、法术、MP、敌人追击和归巢可用。
- 城堡和旧教堂至少一个区域可完整探索。
- UI 显示 HP/MP/装备栏。
- 资源预加载和大模型预热有效。

### 9.3 MVP 可选项

- 更多 NPC 对话。
- 更多武器。
- 更复杂道具系统。
- 更精细音效混音。
- 更多室内敌人和机关。

## 10. 优先级分析

### 10.1 Must Have

- 稳定移动和相机。
- 关键碰撞可靠。
- 一条完整探索路线。
- 基础战斗闭环。
- HP/MP 和死亡复活。
- 城堡或废墟中的至少一个重点探索空间。
- 可接受性能。

### 10.2 Should Have

- 城堡电梯和屋顶体验。
- 巡逻敌人。
- 装备栏和武器切换。
- 加载预热。
- 更统一的黑暗奇幻光照。

### 10.3 Could Have

- 更多武器、道具和技能。
- 更多 NPC 文案和任务。
- 更复杂敌人绕障。
- 可配置画质选项。

### 10.4 Won't Have In First Stage

- 完整剧情。
- 商业级关卡量。
- 联机。
- 存档系统。
- 多端完整适配。

## 11. 关键风险

### 11.1 技术风险

| 风险 | 影响 | 当前迹象 | 缓解建议 |
|---|---|---|---|
| 大模型性能 | 转镜头卡顿、首次渲染卡顿 | 旧教堂 80MB GLB 已出现卡顿 | 模型优化、预加载、预渲染、拆分资源 |
| 碰撞不准 | 穿模、卡住、掉落 | 城堡、屋顶、电梯、废墟多次出现 | 碰撞导出规范化、采样验证、关键区域测试 |
| 状态复杂 | 死亡/复活/场景切换残留 bug | 室外、城堡、室内、敌人逻辑分支多 | 明确状态机和重置规则 |
| 动画与判定不同步 | 攻击手感差 | 敌人/玩家攻击窗口多次调整 | 建立攻击窗口调试和配置化 |
| 资产版本缓存 | 修改后游戏仍加载旧资源 | 城堡多次需要版本号更新 | 统一资源版本和导出流程 |

### 11.2 产品风险

| 风险 | 影响 | 缓解建议 |
|---|---|---|
| 目标从休闲等轴测变成黑暗动作探索，定位漂移 | 需求不聚焦 | 更新 README 和项目定位 |
| 过早扩内容 | 开发失控 | 先交付垂直切片 |
| 视觉暗但不可读 | 玩家迷路或看不清 | 建立光照验收截图 |
| 黑魂参考过重 | 期望超出现有资源能力 | 明确“参考氛围和结构”，不是复刻 |

### 11.3 内容生产风险

| 风险 | 影响 | 缓解建议 |
|---|---|---|
| Blender 手工改动和脚本生成冲突 | 资产回归 | 明确 master blend 和脚本职责 |
| GLB/FBX 坐标、缩放、朝向不统一 | 模型漂浮、反向、握持错位 | 建立资产导入规范 |
| 碰撞对象命名不规范 | 导出错误 | 命名约定文档化 |

## 12. 依赖与约束

### 12.1 技术依赖

- Three.js `^0.169.0`
- Vite `^5.4.0`
- Blender 5.x 本地应用
- Chrome/现代浏览器 WebGL 环境

### 12.2 资源依赖

- 玩家 FBX 与动画文件。
- 敌人 FBX 与动画文件。
- 城堡 Blender 主文件和导出 GLB。
- 旧教堂 GLB 与碰撞 JSON。
- PBR 贴图和音频资源。

### 12.3 工程约束

- 项目使用原生 JS/ESM，不使用 TypeScript。
- 没有成熟测试框架，主要依赖构建、浏览器检查和手动/脚本化场景验证。
- 大部分需求来自快速迭代，对需求基线和验收标准仍需补齐。

## 13. 验收策略

### 13.1 技术验收

- `npm run build` 必须通过。
- 关键页面在本地 dev server 无控制台错误。
- 核心资源可加载，无 404。
- 大模型区域首次转向无明显不可接受卡顿。

### 13.2 玩法验收

- 玩家可完成 MVP 路线。
- 敌人追击、攻击、归巢、重新激活可复现。
- 火球消耗 MP，MP 不足阻止施法。
- 死亡淡出和复活流程正常。

### 13.3 场景验收

- 城堡入口、白雾门、电梯、屋顶、出口可用。
- 旧教堂可进入，主要墙体阻挡正确。
- 曲线峭壁和假山不阻断主路线。

### 13.4 视觉验收

- 室外亮处不过曝。
- 暗处可读。
- 城堡、废墟、峡谷风格基本统一。
- UI 不遮挡核心操作和视野。

## 15. 开放问题

以下问题需要在正式立项前确认：

1. 项目最终目标是个人原型、作品集 Demo，还是可持续开发的完整游戏？
2. 第一阶段是否只面向桌面浏览器？
3. 是否需要存档系统，还是每次刷新都允许重置？
4. 城堡、废墟、峡谷中哪个是第一阶段主关卡？
5. 战斗目标是轻量 ARPG，还是更接近 Souls-like 的高惩罚动作体验？
6. 是否需要将 Blender 资产命名和碰撞规范单独文档化？
7. 是否接受后续为稳定性引入更强工程结构，例如 TypeScript、状态机或测试框架？

## 16. 立项建议

建议立项，但应限定为“第一阶段垂直切片”，不建议直接立项为完整游戏。

理由：

- 项目已经有足够多可验证的玩法和技术组件。
- 风险集中在可识别领域：碰撞、性能、资产流程、状态管理。
- 若继续按临时需求扩展，会增加返工和不稳定性。
- 通过需求基线和 MVP 范围冻结，可以把当前原型转化为可评审、可展示、可继续投资的版本。

建议下一步：

1. 冻结 MVP 路线。
2. 更新 README 项目定位。
3. 为碰撞、资产导出和性能建立固定验收脚本或检查清单。
4. 暂停新增大功能，优先修复关键体验风险。
