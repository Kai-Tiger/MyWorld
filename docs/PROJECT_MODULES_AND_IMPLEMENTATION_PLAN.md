> 项目：浏览器端黑暗奇幻 3D 动作探索游戏原型  
> 日期：2026-06-18  
> 用途：作为后续开发、重构、需求拆分、验收和协作的项目级参考文档  
> 依据：当前代码结构、`PRE_PROJECT_REQUIREMENTS_ANALYSIS.md`、`docs/requirements/`、资源目录与现有实现状态

## 1. 文档定位

本文将项目拆分为可管理的功能模块，并给出分阶段实施计划。

本文解决的问题：

- 当前项目应该按哪些模块组织？
- 每个模块的职责、边界和依赖是什么？
- 哪些模块已经有雏形，哪些需要补齐？
- 后续开发应该按什么顺序推进？
- 每个阶段的交付物和验收标准是什么？

本文不替代具体需求文档。具体功能变更仍应在 `docs/requirements/` 下沉淀为独立需求文档。

## 3. 模块一：运行入口与游戏状态

### 3.1 模块目标

统一管理游戏启动、资源预加载、场景切换、主循环、死亡/复活、渲染分支。

### 3.2 当前职责

当前 `src/main.js` 承担：

- renderer 初始化。
- loading overlay。
- 模型预加载。
- outdoor/castle/indoor 场景切换。
- 玩家、NPC、敌人创建。
- 日夜光照更新。
- 战斗输入分发。
- 死亡淡出与复活。
- 预渲染预热。

### 3.3 目标边界

入口模块应只负责“编排”，不负责具体系统细节。

建议边界：

- `main.js`：应用启动、系统初始化、主循环调度。
- `gameState`：当前场景、死亡状态、loading 状态、交互锁定状态。
- `sceneController`：切换 outdoor/castle/indoor。
- `runtimeUpdate`：按状态调度玩家、敌人、地图、城堡、UI。

### 3.4 实施计划

1. 短期保持 `main.js` 不大拆，避免引入新 bug。
2. 先给运行状态命名并集中：
   - active scene
   - loading/prewarm
   - player death/fade
   - castle transition
3. 中期将死亡流程、场景切换、预热流程拆成独立函数或小模块。
4. 长期再考虑引入显式状态机。

### 3.5 验收标准

- 新增功能不再直接把复杂逻辑堆进主循环。
- 死亡、复活、进入城堡、离开城堡流程可独立读懂。
- 主循环中每帧调用路径清晰。

## 4. 模块二：玩家控制、动画与相机

### 4.1 模块目标

提供稳定、可调、符合动作探索体验的玩家控制。

### 4.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 输入映射 | 键盘、鼠标、交互按键、战斗按键 |
| 移动 | 平面移动、速度控制、碰撞响应 |
| 跳跃 | jump 动画、重力、落地 |
| 动画 | idle/walk/run/attack/hurt/death/cast/sit |
| 相机 | 第三人称跟随、pitch/yaw、遮挡、动态距离 |
| 玩家状态 | HP、MP、体力、死亡、锁定、施法、攻击 |

### 4.3 当前状态

已有：

- 第三人称相机。
- 负 pitch 动态拉近。
- 玩家 FBX 动画。
- 跳跃动画。
- 两段连击。
- MP 系统。
- 法术施放入口。
- 受击、死亡、复活。

主要问题：

- 玩家状态和动画状态容易互相影响。
- 战斗动作、移动锁定和输入缓冲需要更清晰的规则。
- 相机验收依赖手动体验，需要形成固定检查点。

### 4.4 实施计划

#### P1：稳定当前体验

- 梳理玩家状态优先级：
  - death > hurt > bonfire/rest > attack/cast > jump > move/idle
- 明确哪些状态允许移动：
  - 普通移动允许。
  - 攻击/施法按配置限制。
  - 坐下/死亡禁止。
- 为相机建立固定验收场景：
  - 平地。
  - 城堡入口。
  - 峡谷。
  - 废墟附近。

#### P2：抽离玩家动作规则

- 将攻击、施法、跳跃、受击的状态判断从主循环中收敛到玩家或 combat controller。
- 让主循环只调用“请求动作”，具体能不能执行由玩家状态判断。

#### P3：调试与手感工具

- 增加轻量调试显示：
  - 当前动作。
  - 当前 pitch。
  - 是否锁定。
  - 当前 HP/MP/stamina。

### 4.5 验收标准

- 玩家任何时刻只有一个最高优先级动作状态生效。
- 攻击、施法、跳跃不会互相打断出错。
- 相机不会穿到角色脚底或因遮挡剧烈抖动。

## 5. 模块三：战斗、法术与数值

### 5.1 模块目标

形成轻量但可扩展的动作战斗闭环。

### 5.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 近战 | 攻击输入、连击、命中窗口、伤害 |
| 法术 | MP 消耗、冷却、火球生成、命中 |
| 锁定 | 锁定敌人、解除锁定、目标选择 |
| 体力 | 攻击、防御、移动对体力的影响 |
| 命中反馈 | 受击、hitstop、音效、粒子 |
| 数值配置 | 伤害、范围、冷却、速度、资源消耗 |

### 5.3 当前状态

已有：

- `BALANCE.combat` 和 `BALANCE.spells`。
- 玩家两段连击。
- 第二段攻击强化和前冲。
- 火球粒子、MP 消耗、冷却。
- 敌人受击、仇恨触发。
- 部分音效。

主要问题：

- 动画命中窗口和视觉动作一致性仍需调试。
- 战斗规则散布在玩家、主循环、敌人和 spells 中。
- 缺少统一“伤害事件”结构。

### 5.4 实施计划

#### P1：冻结战斗基础规则

- 固定第一阶段只支持：
  - 两段近战。
  - 一种火球法术。
  - 基础防御。
  - 一个治疗/元素瓶系统。
- 暂不增加更多技能树和武器类型。

#### P2：统一伤害事件

目标事件字段：

```text
source
target
damage
type
hitPosition
knockback
aggro
```

短期可不建立复杂类，只统一调用参数和命名。

#### P3：配置化调优

- 所有攻击范围、窗口、倍率、冷却继续放在 `src/config/balance.js`。
- 禁止在实体代码中散落新战斗数值。

### 5.5 验收标准

- 近战和火球都能稳定命中敌人。
- MP 不足阻止火球。
- 远程命中触发敌人追击。
- 玩家攻击范围和动画表现基本一致。

## 6. 模块四：敌人 AI 与 NPC

### 6.1 模块目标

提供少量但稳定的敌人行为，支撑垂直切片战斗体验。

### 6.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 敌人生成 | 从配置创建敌人 |
| 感知 | 视野/距离触发仇恨 |
| 追击 | 向玩家移动，绕开障碍 |
| 攻击 | 攻击动画、窗口、伤害 |
| 归巢 | 超出半径返回初始点 |
| 巡逻 | 沿点位往返 |
| 非敌对 NPC | 对话、朝向、站位 |

### 6.3 当前状态

已有：

- e1/e2 敌人模型和动画。
- 敌人攻击窗口。
- 追击、归巢、返程重新激活。
- 巡逻敌人。
- 友方 NPC f1/f2 与对话。

主要问题：

- 敌人绕障仍是简单局部判定，不适合复杂迷宫。
- 室外敌人和未来室内敌人需要统一生命周期。
- 敌人状态重置规则需要与死亡/复活绑定。

### 6.4 实施计划

#### P1：稳定现有敌人行为

- 只保留少量敌人用于 MVP：
  - 篝火附近敌人。
  - 巡逻敌人。
  - 废墟/迷宫守卫。
- 确认每类敌人有：
  - 初始状态。
  - 追击。
  - 攻击。
  - 归巢/巡逻恢复。
  - 死亡。

#### P2：统一敌人配置

所有敌人通过配置描述：

```text
model
position
rotation
name
patrol?
behaviorProfile?
```

#### P3：室内敌人预留

- 城堡内敌人暂不大量增加。
- 先定义接口，让 castle scene 可以拥有自己的 enemy list 和 update/reset。

### 6.5 验收标准

- 敌人不会追出无限远。
- 返程中可以重新被激活。
- 巡逻敌人脱战后继续巡逻。
- 玩家死亡复活后敌人状态可重置。

## 7. 模块五：世界、关卡与场景

### 7.1 模块目标

将当前松散地图内容收敛为可体验的垂直切片路线。

### 7.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 室外主地图 | 地形、路线、出生点、火堆、敌人 |
| 峡谷/峭壁 | 空间边界和曲线路线 |
| 假山路线 | 出生点到城堡的曲折动线 |
| 旧教堂废墟 | 可进入探索点 |
| 城堡 | 多层室内主关卡 |
| 普通室内 | 旧房屋室内，可降级维护 |

### 7.3 当前状态

已有：

- 大地图尺寸配置。
- 出生点和多个火堆。
- 曲线峭壁。
- 假山路线。
- 旧教堂废墟。
- 多层城堡和屋顶。
- 城堡电梯。

主要问题：

- `src/scene/map.js` 过大，包含地形、装饰、特殊场景、敌人支持等多种职责。
- 关卡路线尚未以“玩家流程”方式冻结。
- 城堡、废墟、峡谷都在扩展，容易互相抢范围。

### 7.4 实施计划

#### P1：冻结 MVP 路线

建议路线：

```text
出生点 -> 篝火/基础敌人 -> 曲折假山路线 -> 旧教堂废墟 -> 城堡入口 -> 城堡电梯 -> 屋顶
```

第一阶段只保证这条路线体验完整。

#### P2：建立关卡检查清单

每个区域必须检查：

- 是否可到达。
- 是否可离开。
- 是否有穿模/掉落。
- 是否有不可解释阻挡。
- 是否有明显卡顿。
- 是否有足够视觉引导。

#### P3：拆分地图生成职责

中期建议从 `map.js` 拆出：

- `terrainLayout`
- `rockeryLayout`
- `cliffLayout`
- `oldChurchPlacement`
- `campfirePlacement`

先拆数据，再拆代码，避免一次性重构。

### 7.5 验收标准

- MVP 路线可以从头走到尾。
- 每个关键区域有明确功能定位。
- 不新增与 MVP 无关的大区域。

## 8. 模块六：城堡与机关

### 8.1 模块目标

城堡作为第一阶段主探索场景，承担室内探索、多层空间、电梯和雾门体验。

### 8.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 城堡外观 | 室外 facade、入口、雾门 |
| 城堡分区 | gatehouse、atrium、floor、roof 等 |
| manifest | bounds、colliders、sockets、connectors |
| 电梯 | 平台、按钮、状态、可用性 |
| 碰撞 | 楼层、墙体、屋顶、竖井 |
| 过渡 | 进入/离开城堡 |
| 火把与灯光 | 室内可读性和氛围 |

### 8.3 当前状态

已有：

- `castle_master.blend`。
- 分区 GLB。
- `castle-manifest.json`。
- 一楼到屋顶电梯。
- 屋顶拉杆和一楼拉杆逻辑。
- 白雾门。
- 室内火把和照明。

主要问题：

- Blender 手工修改和脚本生成之间需要更强约束。
- 屋顶、楼层、电梯竖井等碰撞曾多次回归。
- 城堡需求较多，容易成为无限扩展点。

### 8.4 实施计划

#### P1：冻结城堡玩法范围

第一阶段城堡只保证：

- 雾门进入。
- 多层可探索。
- 电梯一楼/屋顶直达。
- 屋顶可安全行走。
- 可离开或完成体验闭环。

暂不增加：

- 大量室内敌人。
- 复杂机关谜题。
- 更多楼层。

#### P2：建立城堡导出验收

每次导出后检查：

- manifest 可解析。
- 分区 GLB 存在。
- 电梯 lever 只保留一楼和屋顶。
- 电梯竖井无隐藏 surface。
- 屋顶可踩面有 surface 碰撞。

#### P3：规范 Blender 命名

命名建议：

```text
VIS_*      可见模型
COL_*      碰撞体
SOCKET_*   交互/挂点
ZONE_*     分区集合
```

### 8.5 验收标准

- 玩家可从入口进入城堡。
- 玩家可搭电梯上下。
- 玩家不在屋顶或电梯中掉落/悬空。
- 导出流程可重复，不破坏手工建模成果。

## 9. 模块七：碰撞与物理边界

### 9.1 模块目标

提供可靠、可解释、可验证的玩家与场景碰撞。

### 9.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 基础碰撞 | AABB/圆形/盒体 |
| surface | 地面/楼层高度 |
| ramp | 斜坡/楼梯 |
| rotated box | 废墟等旋转碰撞 |
| dynamic collider | 电梯等移动平台 |
| 调试验证 | 采样检查、空间检查 |

### 9.3 当前状态

已有：

- `src/systems/collision.js`。
- 多类碰撞体。
- 城堡 manifest 碰撞。
- 旧教堂碰撞 JSON。
- 电梯动态 surface。

主要问题：

- 碰撞问题是当前最高风险。
- 可视模型和碰撞模型容易不同步。
- 缺少固定自动化碰撞验收脚本。

### 9.4 实施计划

#### P1：定义关键碰撞验收点

必须覆盖：

- 城堡入口。
- 电梯竖井。
- 屋顶。
- 旧教堂入口和内部中轴。
- 假山路线。
- 曲线峭壁。

#### P2：增加检查脚本

建议建立脚本：

- 读取 GLB mesh bbox。
- 读取 manifest/collider JSON。
- 对关键 surface 区域采样。
- 输出未覆盖点和异常 blocker。

#### P3：碰撞命名和导出规范化

- 对 Blender 碰撞对象做命名规范。
- 只让明确 collision 对象导出为运行时碰撞。
- 避免透明可见对象参与碰撞。

### 9.5 验收标准

- 关键路线无穿模。
- 关键路线无隐藏阻挡。
- 电梯和屋顶碰撞稳定。
- 碰撞回归能被脚本发现。

## 10. 模块八：UI、HUD 与交互反馈

### 10.1 模块目标

提供清晰、不遮挡视野、符合黑暗奇幻风格的 HUD 和交互反馈。

### 10.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 状态 HUD | HP、MP、体力 |

## 11. 模块九：资源加载、性能与渲染

### 11.1 模块目标

### 11.2 子模块划分

| 子模块 | 职责 |

### 11.3 当前状态

- `modelAssets.js` 共享缓存。
- Loading 阶段预加载。
- 旧教堂 medium GLB。

主要问题：

- 贴图和模型资源较多，缺少预算。

### 11.4 实施计划

#### P1：建立资源预算

#### P2：预加载清单配置化

- 城堡、玩家、敌人、武器、废墟都纳入启动阶段。

#### P3：性能基准

固定检查：

- 出生点。
- 看向废墟。
- 看向城堡。
- 城堡室内。

### 11.5 验收标准

- Loading 后首次转向重资产区域不卡死。

## 13. 模块十一：资产生产与工具链

### 13.1 模块目标

让 Blender、GLB、FBX、碰撞、图标等资产流程可重复。

### 13.2 子模块划分

| 子模块 | 职责 |
|---|---|
| 城堡构建 | `build_castle.py`、master blend |
| 当前城堡导出 | `export_current_castle.py` |
| 废墟优化 | `optimize_old_church_ruins.py` |
| 废墟碰撞 | `export_old_church_colliders.py` |
| 图标渲染 | `render_model_icon.py` |
| 地形构建 | souls vertical terrain scripts |

### 13.3 当前问题

- 手工改 blend 后，脚本重建可能覆盖用户修改。
- `.blend1` 备份文件容易造成误解。
- 碰撞和可见模型命名规范需要文档化。
- 导出后需要手动检查版本号或资源缓存。

### 13.4 实施计划

#### P1：资产规范文档

建立 `ASSET_PIPELINE.md`，说明：

- 哪个 blend 是主文件。
- 什么情况下用重建脚本。
- 什么情况下只导出当前 blend。
- 命名规范。
- 碰撞对象规范。

#### P2：导出检查清单

每次导出后检查：

- 目标 GLB 更新时间。
- manifest 关键字段。
- 碰撞覆盖。
- 游戏资源版本。
- dev server 是否加载新资源。

#### P3：自动化脚本整合

长期考虑一个命令完成：

```text
导出 -> manifest 检查 -> 资源版本更新 -> build
```

### 13.5 验收标准

- 资产导出可重复。
- 手工 Blender 修改不会被无意覆盖。
- 关键碰撞不会随导出回归。

## 19. 结论

当前项目已经具备足够的玩法、场景和资产基础，适合进入“垂直切片稳定化”阶段。

后续开发的关键不是继续堆功能，而是：

- 明确模块边界。
- 冻结 MVP 路线。
- 稳定碰撞和性能。
- 收敛战斗与状态逻辑。
- 把资产导出和需求文档制度化。

如果按本文档推进，项目可以从高频实验原型逐步转变为可演示、可评审、可继续扩展的黑暗奇幻浏览器 3D 动作探索项目。
