# 垂直切片与死亡复活计划

## 背景判断

当前项目已经具备第三人称移动、锁定、近战、格挡、火球、敌人 AI、NPC 对话、篝火菜单、城堡进入与室内城堡探索等基础能力。问题不在于缺少单个系统，而在于这些能力还没有被串成一个明确、可验证、可反复试玩的核心流程。

本文件只覆盖两个最高优先级目标：

1. 做一个完整垂直切片。
2. 补上死亡、复活、篝火检查点循环。

不在本阶段处理完整背包、复杂任务系统、更多武器、更多地图编辑器能力、钓鱼扩展、剧情分支和大规模资源优化。

## 总目标

做出一个 5 到 10 分钟的可玩版本，让玩家能从出生点开始，经历 NPC 提示、篝火、战斗、死亡/复活、雾门、城堡探索，并抵达一个明确终点。

成功标准：

- 新玩家不看代码，也能理解自己应该往城堡方向前进。
- 玩家能在路线上遇到至少一场可完成的战斗。
- 玩家死亡后能回到最近激活的篝火，不需要刷新页面。
- 篝火休息能恢复玩家状态，并形成再次挑战的循环。
- 穿过雾门进入城堡后，有一个短目标可以完成。
- 整条流程可以在本地连续试玩三次，不出现卡死、黑屏、无法输入、无法复活或无法退出城堡的问题。

## 第一项：完整垂直切片

### 设计目标

垂直切片不是继续扩大地图，而是把已有系统组织成一条完整体验。它要证明这个游戏的核心方向成立：

- 探索：玩家沿道路、地形和 NPC 提示前往城堡。
- 准备：玩家在篝火处获得休息点和心理安全点。
- 战斗：玩家通过锁定、近战、格挡、火球解决敌人。
- 风险：玩家可能死亡。
- 推进：玩家穿过雾门进入城堡。
- 收束：玩家在城堡内找到一个明确的短目标或出口。

### 玩家流程

推荐第一版流程如下：

1. 出生在主篝火附近。
2. 玩家看见城堡方向、道路方向和两个非敌对 NPC。
3. NPC 台词给出三个信息：城堡危险、篝火能休息、雾门后有目标。
4. 玩家靠近篝火，能打开篝火菜单并休息。
5. 玩家沿路前进，遇到第一组敌人。
6. 玩家通过战斗击败敌人，或者死亡并回到篝火。
7. 玩家抵达雾门，点击“穿过白雾”。
8. 玩家进入城堡。
9. 玩家在城堡内完成一个短目标：例如到达大厅出口、打开一扇门、登上露台或触发终点提示。
10. 玩家看到明确的完成反馈，例如“你已抵达城堡大厅深处”。

### 最小内容范围

只需要一条路线，不需要开放世界规模内容。

必需内容：

- 一个出生点。
- 一个默认检查点篝火。
- 两个 NPC，其中至少一个直接告诉玩家下一步方向。
- 一条通向城堡入口的主路。
- 一组教学敌人，数量建议 1 到 2 个。
- 一个雾门交互。
- 一个城堡内短目标。
- 一个完成状态反馈。

暂不需要：

- 多任务日志。
- 多结局。
- 复杂对话选择。
- 新地图区域。
- 新敌人类型。
- 新武器。
- 大量可拾取物。

### 建议路线布局

基于当前配置，优先复用已有内容：

- 出生点：靠近 `world.js` 里的中心篝火 `[2, -2]`。
- NPC：复用 `main.js` 中已经生成的“异乡防毒面具男人”和“蒙面女弓箭手”。
- 第一战：复用 `world.js` 里的“篝火巡守”和“篝火伏击者”，但要确认数量和距离不会让玩家出生后立刻被围攻。
- 城堡入口：复用现有 `enterCastle()` 和“穿过白雾”交互。
- 城堡短目标：优先复用 `castle.getNearbyAction()` 或 `castle.consumeExitRequest()` 附近已有交互，不新建复杂系统。

### NPC 台词调整方向

NPC 台词不需要长，应该承担引导功能。建议每个 NPC 的首次对话控制在 3 到 4 句。

防毒面具男人负责告诉玩家：

- 前方雾门通向城堡。
- 路上有守卫。
- 篝火可以作为休息点。

蒙面女弓箭手负责告诉玩家：

- 锁定敌人再战斗。
- 火球适合拉开距离。
- 城堡里有一条能继续前进的路。

成功标准：

- 玩家读完任意一个 NPC 对话后，知道自己要去雾门。
- 玩家能从台词中知道篝火和战斗系统的存在。

### 战斗教学目标

第一场战斗只验证现有系统，不追求高难度。

建议目标：

- 敌人数量：第一版最多 2 个。
- 敌人站位：一个明显可见，一个稍后进入战斗。
- 玩家可用手段：近战、格挡、火球、锁定。
- 失败代价：死亡后回篝火。

成功标准：

- 玩家不锁定也可以勉强打赢。
- 玩家使用锁定会明显更容易。
- 火球可以稳定命中一个被锁定的目标。
- 敌人攻击能造成威胁，但不会在玩家刚接触时秒杀。

### 城堡短目标

第一版城堡目标不要做成大迷宫。建议选择下面一个：

方案 A：到达大厅终点。

- 玩家进入城堡后，沿主要通道到达大厅深处。
- 靠近终点触发“已抵达大厅深处”的提示。
- 这是最简单方案。

方案 B：打开出口门。

- 玩家进入城堡后找到一处可交互门。
- 点击后门打开，显示完成提示。
- 适合复用现有城堡门逻辑。

方案 C：登上露台。

- 玩家进入城堡后通过现有路线到达露台。
- 触发露台返回室外或完成提示。
- 如果当前城堡路径已经稳定，这是体验最好的方案。

推荐先做方案 A。它最容易验证，不会被城堡内部碰撞、楼层、电梯问题拖住。

### 代码落点

尽量少改文件。

可能涉及：

- `src/config/world.js`：调整出生附近篝火、敌人站位、敌人数量。
- `src/main.js`：增加垂直切片完成状态、出生点、终点触发检测。
- `src/ui.js`：增加一个简单的完成提示或状态提示。
- `src/entities/npcFBX.js` 或 `src/main.js`：调整 NPC 台词内容。

避免在第一版做：

- 新增任务系统类。
- 重写 UI 框架。
- 重写地图生成。
- 新增复杂存档格式。

### 验收清单

手动验收：

- 从刷新页面开始，玩家出生在主流程附近。
- 玩家能找到 NPC 并完成对话。
- 玩家能看到或推断城堡方向。
- 玩家能打开篝火菜单。
- 玩家能与第一组敌人战斗。
- 玩家能击败敌人并继续前进。
- 玩家能点击雾门进入城堡。
- 玩家能到达城堡短目标并看到完成反馈。
- 玩家能从城堡返回或停留在一个稳定完成状态。

技术验收：

- `npm run build` 通过。
- 控制台没有阻断性运行错误。
- 玩家不会在主流程路线上被碰撞体卡死。
- 战斗后死亡敌人的碰撞体被移除。
- 切换 outdoor/castle/bonfire 状态后输入不会失效。

## 第二项：死亡、复活、篝火检查点循环

### 设计目标

死亡和复活是当前游戏风险循环的核心。没有它，战斗失败只能变成 HP 归零后的半失效状态；有了它，篝火、敌人、城堡推进才会形成可反复挑战的游戏结构。

第一版只做最小循环：

1. 玩家 HP 降到 0。
2. 进入死亡状态，禁用移动和交互。
3. 播放淡出或简单死亡提示。
4. 玩家回到最近激活的篝火。
5. 玩家 HP 恢复。
6. 敌人按规则重置。
7. 游戏恢复 outdoor 状态。

### 检查点规则

推荐第一版规则：

- 游戏开始时默认检查点是离出生点最近的主篝火。
- 玩家在任意篝火选择“休息”后，该篝火成为当前检查点。
- 死亡后回到当前检查点旁边，而不是回到原地。
- 如果没有有效篝火，则回到固定出生点。

检查点数据最小结构：

```js
{
  x: number,
  z: number,
  y: number,
  yaw: number
}
```

暂不需要写入 localStorage。第一版只保证单次运行内有效。

### 玩家死亡状态

触发条件：

- `player.getHp() <= 0`。
- 只触发一次，避免每帧重复进入死亡逻辑。

死亡状态建议命名：

- `activeScene = 'player-dead'`
- 或单独维护 `playerDeathState`

进入死亡状态时：

- 清除锁定目标。
- 隐藏 NPC 血条、对话、篝火菜单、进门提示。
- 禁用第三人称相机输入。
- 停止玩家主动输入处理。
- 保留场景渲染。
- 显示黑屏淡出或死亡文字。

第一版可以没有死亡动画，因为当前 `player.js` 已有受击动画但未看到死亡动画接口。不要为了死亡动画扩大范围。

### 复活流程

推荐流程：

1. 死亡状态开始，记录 `deathStartedAt`。
2. 0 到 1 秒：画面淡出。
3. 1 秒：移动玩家到检查点位置。
4. 恢复 HP 到满值。
5. 清理火球、伤害飘字、锁定目标。
6. 重置或复活敌人。
7. 1 到 2 秒：画面淡入。
8. 切回 `activeScene = 'outdoor'`。

玩家位置：

- `player.setPosition(checkpoint.x, checkpoint.z, checkpoint.y)`
- 之后调用 `thirdPerson.syncToTarget(player.getPosition())`
- 同步 `playerCollidable.x/z`

HP：

- 当前 `player.heal(amount)` 只能加血，没有直接设置满血接口。
- 最小实现可以调用 `player.heal(player.getMaxHp())`。
- 如果后续需要更清晰，可以在 `player.js` 增加 `restoreFullHp()`，但第一版不是必须。

### 篝火休息行为

当前篝火菜单已经有“休息”和坐下/起身状态。第一版应补齐休息的实际效果。

选择“休息”后：

- 当前篝火成为检查点。
- 玩家 HP 恢复满。
- 关闭锁定。
- 清理敌人仇恨。
- 可选择重置敌人。

注意：

- “坐下动画完成”不是核心成功标准。
- 核心成功标准是检查点和恢复效果已经生效。
- 如果动画时序复杂，检查点可以在点击“休息”时立即设置。

### 敌人重置规则

第一版有两种可选方案。

方案 A：死亡后重置所有敌人。

- 最接近魂系篝火逻辑。
- 玩家死亡后所有敌人回到出生点并恢复 HP。
- 需要敌人实体支持复活或重建。

方案 B：死亡后只重置仍活着的敌人位置和仇恨，死亡敌人保持死亡。

- 实现更简单。
- 但循环压力不足，容易变成消耗战。

推荐方案 A，但实现时可以先用“重建敌人列表”的方式，避免给敌人类塞太多复活状态。

当前代码里敌人创建集中在 `main.js`：

- `enemyNpcConfigs.forEach(...)` 创建敌人。
- 敌人加入 `npcs`。
- 敌人碰撞体加入 `collision`。
- 敌人死亡后 `handleNpcDeath()` 会移除碰撞体。

最小重建思路：

1. 把敌人创建逻辑提取为 `spawnEnemyNpcs()`。
2. 保留非敌对 NPC。
3. 复活时移除旧敌人的 group 和碰撞体。
4. 按 `enemyNpcConfigs` 重新创建敌人。
5. 重新加入 `npcs` 和 `collision`。

这比给 `createEnemyNpcFBX()` 增加 `reset()` 更直接，但要小心不要删除非敌对 NPC。

### 需要清理的运行时状态

死亡或篝火休息后，应清理：

- 锁定目标。
- 伤害飘字。
- NPC 血条。
- 敌人仇恨或敌人实例。
- 玩家攻击状态。
- 玩家火球施法 pending 状态。
- 场景里还在飞行的火球。
- 篝火菜单、对话框、进门提示。

当前可能需要补接口：

- `SpellSystem.clear()`：移除所有现存 projectile 和粒子。
- `ui.clearCombatOverlays()` 已存在，可复用。
- `clearLock()` 已存在，可复用。
- `player.playIdle()` 已存在，可复用。

### UI 反馈

第一版只需要两个反馈：

- 死亡提示：例如“你死了”。
- 复活提示：可以省略，画面淡入回到篝火即可。

不要先做复杂死亡菜单。死亡后自动复活即可。

如果要加 UI 接口，建议在 `ui.js` 中增加：

- `showDeathMessage()`
- `hideDeathMessage()`

也可以先复用 `setSceneFade(opacity)`，用纯黑淡入淡出实现。

### 状态机建议

在 `main.js` 里增加一个小状态对象即可：

```js
let checkpoint = null
let deathState = null
```

死亡状态：

```js
deathState = {
  phase: 'fade-out',
  time: 0,
  moved: false
}
```

主循环前段增加：

```js
if (activeScene === 'player-dead') {
  updatePlayerDeath(rawDt)
  return
}
```

触发检测放在 outdoor 战斗更新后、渲染前：

```js
if (player.getHp?.() <= 0) {
  beginPlayerDeath()
  return
}
```

城堡内是否允许死亡：

- 第一版可以统一允许。
- 如果玩家在城堡中死亡，也回 outdoor 检查点。
- 这样最简单，也符合“篝火是安全重试点”的设计。

### 代码落点

尽量集中修改：

- `src/main.js`
  - 当前检查点数据。
  - 篝火休息时设置检查点。
  - 玩家死亡检测。
  - 复活流程。
  - 敌人重建函数。

- `src/entities/player.js`
  - 可选：增加 `restoreFullHp()`。
  - 可选：增加 `isDead()`。

- `src/systems/spells.js`
  - 建议增加 `clear()`，用于死亡/复活时清理火球。

- `src/ui.js`
  - 可选：增加死亡提示。
  - 也可先只使用已有黑屏 fade。

### 实现顺序

1. 定义默认检查点。
   - 验证：刷新页面后 checkpoint 有合法坐标。

2. 篝火休息设置检查点并回满 HP。
   - 验证：受伤后点击休息，HP 回满。

3. 玩家 HP 为 0 时进入 `player-dead`。
   - 验证：敌人打死玩家后，移动和攻击输入不再生效。

4. 死亡后移动玩家回检查点并恢复 HP。
   - 验证：不刷新页面，玩家回到篝火旁。

5. 清理 UI、锁定、火球和战斗状态。
   - 验证：复活后没有残留锁定、血条、伤害飘字或飞行火球。

6. 重置敌人。
   - 验证：被杀死的敌人在玩家死亡后重新出现，碰撞体正常。

7. 连续死亡三次。
   - 验证：不会重复注册敌人碰撞体，不会出现多个相同 UI，不会卡在黑屏。

### 验收清单

手动验收：

- 玩家被敌人攻击到 0 HP 后进入死亡流程。
- 死亡期间玩家不能移动、攻击、开菜单或进门。
- 死亡后自动回到最近休息过的篝火。
- 玩家复活后 HP 满。
- 复活后相机对准玩家，输入恢复。
- 复活后锁定状态为 OFF。
- 复活后敌人状态符合预期重置规则。
- 玩家可以再次挑战同一组敌人。
- 在城堡内死亡也能回到 outdoor 篝火。

技术验收：

- `npm run build` 通过。
- 连续死亡三次后，`collision.collidables` 没有明显重复增长。
- 复活后 `activeScene` 回到 `outdoor`。
- 复活后 `playerCollidable` 与玩家位置一致。
- 复活后没有未清理的 projectile 留在场景中。

## 推荐第一版交付拆分

第一次提交：

- 垂直切片路线固定。
- NPC 台词缩短并服务引导。
- 城堡内短目标和完成提示。

第二次提交：

- 检查点数据。
- 篝火休息回满 HP。
- 玩家死亡自动回篝火。

第三次提交：

- 敌人重置。
- 火球和战斗 UI 清理。
- 连续死亡稳定性检查。

这样拆分的原因是每次提交都有独立可验收结果，且不会一次性改太多系统。

## 暂缓事项

下面内容容易拖慢进度，建议等上述流程稳定后再做：

- 新增完整任务日志。
- 新增多种篝火功能。
- 加入消耗品和药瓶数量。
- 大规模重做 UI 视觉。
- 继续扩展城堡楼层。
- 增加新武器和新敌人。
- 做正式存档。
- 做资源压缩和分包。

这些都重要，但不是让当前版本变成“可玩游戏”的第一步。
